The Problem

Writing a Statement of Work is one of the most time-consuming parts of the presales process. A typical SOW touches compliance requirements, project scope, acceptance criteria, timelines, and budget — all of which vary by client, industry, and regulatory environment. Done manually, it takes hours. Done poorly, it creates legal exposure.

The client needed a system that could generate a complete, grounded SOW in seconds — given just a client name and some deal context. The hard constraint: no hallucinated clause numbers, no invented compliance references, no missing sections. Every output had to be traceable to a real source document.

Without RAG, the LLM confidently produced invented HIPAA clause numbers and missing acceptance criteria. With RAG grounded on actual regulatory text, the output became structurally correct and source-traceable on the first attempt.

Agent Workflow

The agent is a LangGraph StateGraph — four nodes, linear chain, shared SOWState TypedDict that each node reads and partially updates.

01
parse_input — Normalises keys, fills defaults, auto-infers compliance tags from keywords ("health" → HIPAA, "payment" → PCI-DSS). Everything downstream depends on a clean deal dict.
02
retrieve_context — Fires three FAISS queries targeting SOW structure, compliance requirements, and AWS best practices. Retrieves k=2 chunks each, deduplicates by content fingerprint. Index loads once per process and stays cached.
03
generate_sow — Builds a structured prompt with deal data + retrieved chunks + explicit section instructions, then calls llama-3.1-8b-instant. Always has complete, grounded input before generating.
04
validate_output — Checks all five required sections are present, word count is over 300, and client name appears in output. Non-blocking — returns warnings alongside the SOW so users can judge the output themselves.
RAG Implementation

The knowledge base has three documents: a full SOW template, the actual HIPAA security rule text (45 CFR Parts 160/164), and an AWS migration best-practices guide — 994 lines total, split into 166 chunks using RecursiveCharacterTextSplitter at 500 characters with 50-character overlap.

Chunking splits on paragraph breaks before line breaks before spaces, keeping chunks semantically coherent. Embeddings run locally via all-MiniLM-L6-v2 — no API cost, sub-second latency after warmup. The flat FAISS index handles brute-force O(n) search efficiently at this scale.

Key Design Decisions

Why Groq? Llama 3.1 8B on Groq delivers near-instant inference — critical for a presales tool where the sales team is often on a live client call. Waiting 30 seconds for an LLM response kills the demo.

Why FAISS over a vector DB? At 166 chunks, a managed vector database adds operational overhead with no performance benefit. FAISS runs in-process, persists to disk, and loads in milliseconds. The right tool for the scale.

Why LangGraph over a simple chain? The four-node StateGraph makes each stage independently testable, observable, and replaceable. When the client asked to add a compliance-check node later, it was a single graph modification — not a rewrite.

Tradeoffs & What's Next

The FAISS index and embedding model are module-level singletons — fine for a single worker but each additional uvicorn worker loads its own ~100MB model copy. A real production deployment would centralise the embedding service. There's currently no auth on the API, no streaming, and no feedback loop for improving retrieval quality over time — all intentional scope cuts for the initial client delivery.

The natural next step is a document ingestion pipeline handling PDFs, DOCX, and Confluence pages — replacing the three static .txt files with a live knowledge base that updates as the client's templates and compliance requirements evolve.
