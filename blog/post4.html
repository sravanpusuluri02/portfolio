Agentic AI in Production — Lessons from Building Multi-Agent Systems That Actually Work

Demos are easy. Production is where agents go to die. Here's what I learned the hard way building a multi-agent RAG system handling 500+ concurrent requests at MetLife.

The demo worked perfectly. Production did not.

I remember the first time our multi-agent system hallucinated its way through a compliance query in a staging environment, confidently citing a policy document that didn't exist. The agent had retrieved three real documents, synthesized a plausible-sounding answer, and invented a fourth source to back it up. The reasoning chain looked completely coherent. The output was completely wrong.

That was week two of building. It took another six weeks to ship something I'd trust in production. Here's what those six weeks taught me.

Why agent loops fail in production

The ReAct pattern — Reason, Act, Observe, repeat — looks elegant in a Jupyter notebook. In production, it has failure modes that demos never hit.

The most common one is agent loops. An agent gets stuck in a reasoning cycle, calling the same tool repeatedly with slightly different inputs, never converging on an answer. In a notebook this just runs forever. In production, it burns tokens, blocks the response queue, and drives up your API costs. We hit this in the first week. Our fix was strict loop detection in LangGraph — a hard limit on node revisits with a graceful fallback response when the limit is reached.

The second failure mode is tool calling confidence. LLMs are optimistic about tool calls. They'll call a search tool with a query so specific it returns zero results, then either hallucinate results or give up entirely. We solved this by implementing a fallback retrieval strategy — if a specific query returns fewer than 3 results, automatically broaden the query and retry with semantic similarity only.

Every failure mode we hit in production was something the demo never showed us. Build your evaluation suite before you build your agent. Not after.

The architecture that actually works

Our final architecture uses LangGraph with the ReAct pattern, but with significant guardrails built into the graph itself rather than relying on prompt engineering alone.

The graph has five nodes: Query Analyzer (classifies intent and routes to appropriate retrieval strategy), Retriever (hybrid semantic + BM25 search against ChromaDB), Context Evaluator (checks retrieved context relevance before passing to generator — this alone eliminated 40% of hallucinations), Generator (Amazon Bedrock with Llama 3), and Response Validator (checks output against retrieved sources using a lightweight verification model).

The Context Evaluator node was the key insight. Most RAG pipelines retrieve context and immediately pass it to the generator. We added an explicit evaluation step: does the retrieved context actually answer the question? If relevance score falls below threshold, we trigger a second retrieval pass with a reformulated query before generating. This added ~120ms to average latency but reduced hallucination rate significantly.

Observability is not optional

You cannot debug an agent system you cannot observe. This sounds obvious. It took me an embarrassingly long time to take it seriously.

We integrated LangSmith early in development and it transformed how we worked. Every agent run creates a complete trace — every LLM call, every tool call, every retrieval result, every decision point. When a user reports a bad response, I can pull up the exact trace, see exactly what the agent retrieved, what it reasoned about that context, and where the reasoning went wrong. Debugging went from "reproduce the issue and hope" to "look at the trace."

We also built automated RAGAS evaluation into our CI/CD pipeline. Every deployment triggers a run against our golden dataset — 500 curated query-answer pairs. If faithfulness score drops below 88% or answer relevancy drops below 85%, the deployment is blocked. This caught three regressions before they hit production.

AI Guardrails in a regulated environment

MetLife isn't a startup. We operate under financial regulations, data privacy requirements, and internal compliance frameworks that don't care how impressive your agent architecture is. Amazon Bedrock Guardrails became a non-negotiable part of our stack.

We configured guardrails for PII redaction (no customer PII should appear in agent responses), content filtering (no speculative financial advice), and denied topics (anything outside the defined scope of the agent gets a graceful refusal). The guardrails layer sits between the generator and the response validator, adding roughly 30ms to response time but providing the compliance team with the audit trail they need.

What I'd tell myself on day one

Build your evaluation suite before you write your first agent node. Know what "good" looks like before you start building. Define your failure modes on paper and build explicit handling for each one into the graph — don't rely on prompting the LLM to avoid them. Instrument everything with LangSmith from day one, not as an afterthought. And accept that the first version will fail in ways the demo never predicted. That's not a bug in your process. That's just production.
