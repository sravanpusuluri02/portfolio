Building an AI SOW Agent for a Real Client — What Actually Worked

When a client asked me to automate their Statement of Work generation, I thought it would be a straightforward RAG project. It wasn't. Here's what I learned building an agentic system that had to be right every single time.

The brief

The client was a B2B SaaS company with a small presales team. Their problem was simple: writing a Statement of Work took hours, happened late in the sales cycle, and was often done under pressure. The output was inconsistent — some SOWs were thorough, others missed acceptance criteria entirely. A few had invented compliance clause numbers that legal had to quietly fix before sending.

What they wanted: give the system a client name and some deal context, get back a complete, structured SOW in seconds. The constraint that made it non-trivial: every compliance reference had to be real. No hallucinated HIPAA clause numbers. No invented AWS architecture recommendations. Everything traceable to an actual source document.

That constraint ruled out a simple "just prompt GPT-4" approach immediately. And it's what made building this genuinely interesting.

Why RAG wasn't optional

I ran the experiment before committing to the architecture. A vanilla LLM prompt — give it the deal context, ask for a SOW — produces output that looks impressive on first read. Structured sections, professional language, reasonable timelines. But look closer and you find invented clause numbers, hallucinated AWS service names, and acceptance criteria that don't match the project scope.

The LLM doesn't know it's hallucinating. It produces confident, well-formatted nonsense. That's worse than an obviously wrong answer — it's a wrong answer that gets sent to the client.

The fix was grounding. I built a knowledge base from three real documents: the client's internal SOW template, the actual HIPAA security rule text (45 CFR Parts 160/164), and an AWS migration best-practices guide. 994 lines total, chunked into 166 pieces using RecursiveCharacterTextSplitter at 500 characters with 50-character overlap.

The overlap matters more than people think. Without it, a chunk that ends mid-sentence loses its context. With 50-character overlap, each chunk carries enough trailing context that the embedding captures the right semantic neighbourhood. The difference in retrieval quality is measurable.

Why I chose LangGraph over a simple chain

My first instinct was a LangChain sequential chain. Parse input → retrieve context → generate → done. It worked in testing. Then the client asked: "Can we add a compliance check step that flags if a deal is missing required HIPAA clauses before we generate the SOW?"

With a sequential chain, that's a rewrite. With LangGraph, it's adding a node and an edge. That conversation — happening two weeks into the project — validated the architecture decision retroactively.

The four-node StateGraph I ended up with: parse_input → retrieve_context → generate_sow → validate_output. Each node reads from and writes to a shared SOWState TypedDict. Each node is independently testable. Each node can be swapped or extended without touching the others.

The validate node is particularly important. It checks that all five required sections are present, word count exceeds 300, and the client name appears in the output. Crucially, it's non-blocking — it returns warnings alongside the SOW rather than raising an exception. The presales team sees the validation result and makes the call. That's the right boundary between the system and the human.

The retrieval design that actually worked

Single-query RAG was not good enough. A query like "generate SOW for healthcare client" retrieves chunks about SOW structure but misses the HIPAA compliance requirements and the AWS architecture guidance. Three separate queries — one for SOW structure, one for compliance requirements, one for technical best practices — and deduplicate by content fingerprint before injecting into the prompt.

The deduplication step sounds minor but it matters. Without it, the same chunk can appear multiple times in the context window, wasting tokens and biasing the generation toward whatever that chunk says. A simple content hash comparison eliminates duplicates in microseconds.

Embeddings run locally via all-MiniLM-L6-v2. No API cost, sub-second latency after warmup, and the model stays in-process. For a knowledge base this size, a managed vector database would add operational overhead with no performance benefit. FAISS in-process is the right call at 166 chunks.

What I'd do differently

The FAISS index and embedding model are module-level singletons. That's fine for a single uvicorn worker, but if you scale to multiple workers each process loads its own ~100MB model copy. A centralised embedding service would be the right fix for production scale.

The knowledge base is three static .txt files. That was the right call for the initial delivery — get something working and validated before building a document ingestion pipeline. But it's the first thing to address in the next phase. The client's SOW template changes. New compliance requirements emerge. Static files don't age well.

The thing I'm most satisfied with: the validate node. Making it non-blocking was a deliberate design decision that pushed back against the instinct to make the system fully autonomous. A presales SOW going to a client is a legal document. The system should flag problems and present them clearly — not silently fail or autonomously decide the output is good enough. The human stays in the loop on the final call.

That's not a limitation. That's responsible AI design.
